## 🎮 معرفی پروژه

**Simple Doodle Jump** یک بازسازی کامل از بازی موبایلی محبوب *Doodle Jump* است که به زبان **C++17** و با استفاده از کتابخانه‌ی گرافیکی **SFML 2.6** پیاده‌سازی شده است.

این پروژه در قالب تمرین درس **برنامه‌نویسی پیشرفته (Advanced Programming)** نوشته شده و تمرکز اصلی آن بر رعایت دقیق اصول زیر است:

- شی‌گرایی (Object-Oriented Programming)
- چندریختی (Polymorphism) از طریق dynamic binding
- اصول Clean Code و SRP (تک‌مسئولیتی)
- مدیریت صحیح حافظه بدون نشتی (Memory Leak)

> 💡 بازیکن به‌صورت خودکار می‌پرد؛ شما فقط با کلیدهای **← / →** او را چپ و راست هدایت می‌کنید و تا جای ممکن بالا می‌روید. افتادن به پایین صفحه به معنای پایان بازی است.

| نوع سکو | رفتار |

## 🧱 معماری و طراحی

پروژه کاملاً شی‌گرا و بر پایه‌ی چندریختی از طریق **اشاره‌گر به کلاس پایه + متدهای virtual** ساخته شده است. رفتار متفاوت هر سکو با **dynamic binding** حل می‌شود، نه با شرط‌های `if/else` پراکنده در کد.

### سلسله‌مراتب کلاس‌ها

```
Entity  (abstract: update / draw / getBounds)
├── Player
├── Spring
└── Platform  (abstract: onLand / isSupportive / supportVelocity)
    ├── NormalPlatform     (می‌تواند یک Spring داشته باشد)
    ├── MovingPlatform
    └── BreakablePlatform

State  (abstract: handleEvent / update / render)
├── MenuState
├── PlayState
└── GameOverState
```

### تصمیم‌های کلیدی طراحی

| اصل طراحی | نحوه پیاده‌سازی |
|---|---|
| **تک‌مسئولیتی (SRP)** | `Player` فقط حرکت/فیزیک خودش، `CollisionSystem` فقط برخوردها، `PlatformManager` فقط تولید/حذف سکوها، `Game` فقط حلقه اصلی و منابع |
| **برخورد متمرکز** | همه‌ی سکوها از یک منطق برخورد واحد عبور می‌کنند؛ پاسخ متفاوت با `Platform::onLand()` به‌صورت چندریخت واگذار می‌شود (بدون تکرار کد) |
| **نگهداری چندریخت** | استفاده از `std::vector<Platform*>` به‌جای `vector<Platform>` برای حفظ چندریختی |
| **فیزیک مستقل از فریم‌ریت** | همه‌ی حرکت‌ها در `dt` ضرب می‌شوند؛ `dt` برای پایداری کلمپ (clamp) می‌شود |
| **بدون Magic Number** | تمام مقادیر قابل‌تنظیم در `include/core/Constants.hpp` (namespace `cfg`) متمرکز شده‌اند |
| **دوربین پویا** | یک `sf::View` عمودی که فقط رو به بالا حرکت می‌کند؛ بازیکن در فضای دنیا بالا می‌رود و دنیا زیرش اسکرول می‌شود |

### تضمین دسترسی‌پذیری (Accessibility)

برای اینکه بازی همیشه قابل ادامه باشد، دو قانون ریاضی/منطقی رعایت شده:

1. حداکثر فاصله‌ی عمودی بین دو سکو `MAX_GAP = 95px` است؛ و `2 × 95 = 190px` کمتر از حداکثر ارتفاع پرش معمولی (`≈ 225px`) است.
2. هیچ‌گاه دو سکوی شکستنی پشت‌سرهم قرار نمی‌گیرند — نوع سکوی قبلی با `dynamic_cast` تشخیص داده می‌شود.

**نتیجه:** بازی همواره از نظر ریاضی قابل‌ادامه است و هیچ نقطه‌ی بن‌بستی برای بازیکن ایجاد نمی‌شود.

## ✅ تطبیق با الزامات پروژه

این فاز دقیقاً طبق چک‌لیست الزامات درسی پیاده‌سازی شده است:

| الزام | وضعیت | جزئیات در کد |
|---|---|---|
| منطق کامل در کلاس‌ها (OOP) | ✅ | `main` فقط `Game` را می‌سازد و اجرا می‌کند |
| سطح دسترسی اعضا (`private`/`protected`) | ✅ | هیچ عضو داده‌ی `public` وجود ندارد |
| `main` کوتاه و غیررویه‌ای | ✅ | `src/main.cpp` |
| **Template معنادار** | ✅ | `ResourceManager<T>` با `sf::Texture` و `sf::Font` — بارگذاری/کش یک‌باره‌ی دارایی‌ها |
| **`static_cast`** برای تبدیل عددی | ✅ | در محاسبات فیزیک، چیدمان و امتیاز |
| **`dynamic_cast`** برای تشخیص نوع سکو | ✅ | `PlatformManager::lastPlatformIsBreakable()` |
| پرهیز از cast سبک C | ✅ | هیچ cast سبک C در کد نیست |
| مدیریت حافظه (بدون leak) | ✅ | هر `new` یک `delete` دارد + `unique_ptr` |
| ساختار چندفایلی (`.hpp`/`.cpp`) | ✅ | تفکیک کامل `include/` و `src/` |
| طراحی توسعه‌پذیر (دید بلندمدت) | ✅ | افزودن ویژگی جدید = افزودن یک کلاس فرزند |
| Clean Code | ✅ | نام‌گذاری استاندارد، توابع کوچک، بدون Magic Number، اصل DRY |

### 🧩 قالب معنادار (Template)

`ResourceManager<Resource>` یک کش عمومی برای هر دارایی‌ای است که متد `loadFromFile` دارد. هر فایل فقط یک‌بار از دیسک خوانده می‌شود. در حال حاضر با `sf::Texture` و `sf::Font` استفاده می‌شود و در فاز بعد مستقیماً برای `sf::SoundBuffer` نیز قابل‌استفاده خواهد بود.

### 🔄 تبدیل نوع (Type Casting)

| نوع Cast | کاربرد در پروژه |
|---|---|
| `static_cast` | تمام تبدیل‌های عددی (مانند `int` به `float`) |
| `dynamic_cast` | در `PlatformManager::lastPlatformIsBreakable()` برای تشخیص اینکه آخرین سکوی ساخته‌شده از نوع `BreakablePlatform` است یا نه (اجرای قانون «دو سکوی شکستنی پشت‌سرهم نباشد») |

### 🧠 مدیریت حافظه

- سکوها با `new` ساخته می‌شوند و در `PlatformManager::removeExpired` (هنگام خروج از صفحه) و در `clear()` (در مخرب کلاس) با `delete` آزاد می‌شوند.
- فنرها، State فعال، و منابع با `std::unique_ptr` مدیریت می‌شوند (آزادسازی خودکار طبق اصل RAII).

## 📁 ساختار پروژه

کد بر اساس **کارکرد** در زیرپوشه‌های آینه‌ای `include/` و `src/` سازمان‌دهی شده است:

```
simple-doodle-jump/
├── assets/                     # تصاویر بازی (پس‌زمینه، دودل، سکوها، فنر، دکمه‌ها)
├── fonts/                      # فونت (ariblk.ttf)
├── include/
│   ├── core/       (Constants, Game, ResourceManager<T>)
│   ├── entities/   (Entity, Player, Spring, platforms/*)
│   ├── world/      (CollisionSystem, PlatformManager)
│   ├── ui/         (Button, Layout)
│   ├── states/     (State, StateId, Menu/Play/GameOver)
│   └── utils/      (Random)
├── src/                        # پیاده‌سازی‌ها (آینه‌ی include/)
├── makefile                    # ساخت چندسکویی (Windows / Linux / macOS)
├── highscore.txt               # بالاترین امتیاز ذخیره‌شده
├── .gitignore
└── README.md
```

> 📌 بدنه‌ی متدها همیشه در فایل `.cpp` نوشته شده تا فایل‌های هدر تمیز و خوانا بمانند.

## ▶️ ساخت و اجرا

### 🐧 لینوکس (Ubuntu / Debian)

```bash
sudo apt update
sudo apt install g++ make libsfml-dev
make
make run
```

### 🍎 macOS (Homebrew)

```bash
brew install sfml@2
# روی Apple Silicon:
make SFML_INC=-I/opt/homebrew/opt/sfml@2/include SFML_LIB=-L/opt/homebrew/opt/sfml@2/lib
./simple-doodle-jump
```

### 🪟 ویندوز (MinGW-w64 + Git Bash)

1. کامپایلر **GCC** (WinLibs 64-bit) و **SFML 2.6.1 هم‌نسخه با کامپایلر** (GCC MinGW SEH 64-bit) را نصب کنید — نسخه‌ی کامپایلر و SFML باید یکسان باشند.
2. در **Git Bash**، از ریشه‌ی پروژه (مسیر SFML را مطابق نصب خودتان تنظیم کنید):

```bash
mingw32-make SFML_INC=-IC:/SFML/include SFML_LIB=-LC:/SFML/lib
```

> اگر SFML در مسیری با فاصله نصب شده (مانند `C:\Program Files\SFML-2.6.1`)، از مسیر کوتاه استفاده کنید تا فاصله مشکل ایجاد نکند:
> ```bash
> mingw32-make "SFML_INC=-IC:/PROGRA~1/SFML-2.6.1/include" "SFML_LIB=-LC:/PROGRA~1/SFML-2.6.1/lib"
> ```

3. فایل‌های `*.dll` داخل `C:\SFML\bin` را کنار `simple-doodle-jump.exe` کپی کنید (یا مسیر `C:\SFML\bin` را به PATH اضافه کنید).
4. اجرا از ریشه‌ی پروژه:

```bash
./simple-doodle-jump.exe
```

### دستورهای makefile

| دستور | کار |
|---|---|
| `make` | ساخت بازی |
| `make run` | ساخت و اجرا |
| `make clean` | پاک‌کردن خروجی‌های ساخت |

> 📌 بازی را حتماً از **ریشه‌ی پروژه** اجرا کنید؛ مسیرهای `assets/`، `fonts/` و `highscore.txt` به‌صورت نسبی خوانده می‌شوند.

## 🧼 استاندارد کد و نام‌گذاری

این پروژه از یک گایدلاین نام‌گذاری واحد پیروی می‌کند که در ابتدای هر فایل نیز اعلام شده است:

| دسته | قاعده | نمونه |
|---|---|---|
| کلاس‌ها / تایپ‌ها | `PascalCase` | `PlatformManager` |
| توابع / متدها | `camelCase` | `handleInput` |
| متغیرها و پارامترها | `lower_snake_case` | `world_top` |
| متغیرهای عضو | `lower_snake_case_` (زیرخط انتهایی) | `high_score_` |
| ثابت‌ها | `UPPER_SNAKE_CASE` | `MOVE_SPEED` |

**اصول رعایت‌شده در سراسر کد:**

- توابع کوچک و تک‌منظوره
- `main` کوتاه و بدون منطق پیچیده
- عدم استفاده از Magic Number
- رعایت اصل DRY (Don't Repeat Yourself)
- کامنت‌های «چرایی» به‌جای توضیح بدیهیات
- توابع `const` برای متدهای فقط‌خواندنی
- پیاده‌سازی متدها بیرون از تعریف کلاس (جداسازی هدر و پیاده‌سازی)

## 🧯 مشکلات رایج (Troubleshooting)

| مشکل | راه‌حل احتمالی |
